## Agent Integration (Quick Install)

Wire an AI agent (Claude Code, Codex, Gemini CLI, GitHub Copilot, Kiro, opencode)
to a running skopos with one command — it merges the MCP server config and copies
the steering/skill files for you:

```bash
# Local skopos (default: http://localhost:8080/mcp), global config
skopos install --agent kiro

# Remote skopos with an API key
skopos install --agent claude-code \
  --url https://skopos.example.com/mcp \
  --api-key "$SKOPOS_API_KEY"

# Install into the current project (instead of ~/.config) and preview first
skopos install --agent kiro --scope project --dry-run

# All supported agents at once
skopos install --agent all --api-key "$SKOPOS_API_KEY"
```

Flags: `--agent` (required), `--url` (default `http://localhost:8080/mcp`),
`--api-key` (env `SKOPOS_API_KEY`; sent as `Authorization: Bearer`), `--scope`
(`global` default | `project`), `--dry-run`. Existing config is backed up to
`<file>.skopos.bak` and the run is idempotent. Per-agent manual steps are in
[`docs/integrations/`](docs/integrations/).

`skopos install` wires the **MCP connection**. To also equip an agent with
skopos **skills/commands/rules** (load context, report status, remember
findings, always-on guidance) compiled from one canonical source into every
target's native format, use the bundled [agent-builder artifacts](docs/integrations/agent-builder/README.md)
(`agent-builder compile docs/integrations/agent-builder/ab-src && agent-builder install`).

## Configuration

Configuration is loaded from `skopos-config.toml` (gitignored) and can be overridden via environment variables. Copy the committed template to create yours:

```bash
cp skopos-config.example.toml skopos-config.toml
```

If `skopos-config.toml` is absent, skopos runs on flag defaults + environment variables, so the file is optional.

Key configuration options:

| Option | Env Var | Default | Description |
|--------|---------|---------|-------------|
| `log.level` | `LOG_LEVEL` | `info` | Log level (debug, info, warn, error) |
| `log.format` | `LOG_FORMAT` | `text` | Log format (text, json) |
| `server.host` | `SERVER_HOST` | `0.0.0.0` | HTTP server host |
| `server.port` | `SERVER_PORT` | `8080` | HTTP server port |

## SRV Record Support

This application supports DNS SRV record discovery for service location. When connecting to databases or cache services, you can use either:

### Direct Connection
```
host:port
```
Example: `localhost:5432`, `redis.example.com:6379`

### SRV Discovery
```
_service._proto.hostname
```
Example: `_postgresql._tcp.db.cluster`, `_redis._tcp.cache.cluster`

The application will automatically detect the format:
- If the host starts with `_`, it's treated as an SRV name and resolved via DNS
- Otherwise, it's parsed as a direct `host:port` connection

### Database Connection

```go
import "github.com/martinsuchenak/skopos/internal/db"

// Open the SQLite database (foreign keys + WAL are configured per connection).
sqlDB, err := db.Connect(log, "skopos.db")
```

`db.Connect` returns a `*sql.DB` configured for SQLite (WAL journal mode,
foreign keys enabled per connection, and a bounded connection pool). Migrations
are applied separately via `db.RunMigrations(sqlDB)`.

### Valkey Connection

```go
import "github.com/martinsuchenak/skopos/internal/valkey"

// Using SRV discovery
client, err := valkey.NewClient(log, "_valkey._tcp.cache.cluster", password)

// Using direct connection
client, err := valkey.NewClient(log, "localhost:6379", password)
```
